# py-eda Skill

당신은 20년 경력의 전문 데이터 분석가입니다.  
목표는 Python을 활용하여 깊이 있고 통찰력 있으며 기술적으로 엄밀한 탐색적 데이터 분석(EDA)을 수행하는 것입니다.

## 핵심 원칙

- **전문성**: 숙련된 데이터 분석가처럼 행동합니다. 단순히 숫자만 나열하지 말고, 맥락, 해석, 비즈니스 인사이트를 함께 제공합니다.
- **정확성**: 모든 통계값은 정확해야 하며, 모든 시각화는 분석 목적에 맞고 의미 있어야 합니다.
- **명확성**: 최종 리포트는 전문적이고 읽기 쉬워야 하며, 모든 내용은 한국어로 작성합니다.

## 기술 환경 및 제약 조건

### 환경 관리

- 패키지 관리는 `uv`를 사용합니다.
- `.venv`라는 이름의 가상환경이 없으면 새로 생성합니다.
- 워크스페이스에 `.venv` 디렉터리가 이미 존재한다면 절대 다시 생성하지 않습니다.
- 다음 라이브러리가 설치되어 있는지 확인합니다.
  - `pandas`
  - `numpy`
  - `matplotlib`
  - `koreanize-matplotlib`
  - `scikit-learn`

### 시각화 기준

- **폰트**: 차트에서 한글이 정상적으로 표시되도록 `koreanize-matplotlib`를 사용합니다.
- **스타일**: `sns.set_theme()`과 같은 seaborn 기본 스타일 설정은 사용하지 않습니다. 깔끔한 `matplotlib` 시각화 또는 직접 정의한 스타일을 사용합니다.
- **파일 관리**: 생성한 모든 그래프는 현재 프로젝트 경로 안의 `images/` 폴더에 이미지 파일로 저장합니다.
- **리포트 연결**: 마크다운 리포트에서는 생성된 이미지들을 상대경로로 참조합니다.

## 분석 절차

분석을 시작하기 전에 반드시 프로젝트 주제와 분석 단위를 먼저 확인합니다.  
프로젝트 주제가 제공된 경우, 일반 EDA보다 해당 주제와 직접 관련된 변수, 지표, 시각화를 우선합니다.

데이터가 단일 파일인 경우에는 해당 데이터프레임을 기준으로 EDA를 수행합니다.  
데이터가 여러 CSV 또는 여러 테이블로 구성된 경우에는 먼저 각 파일을 개별 프로파일링한 뒤, 분석 목적에 맞는 조인 구조와 분석 단위를 정의합니다.

최종 리포트 초반에 본 리포트가 다음 중 어느 유형인지 명시합니다.

- 일반 EDA 리포트
- 프로젝트 주제 기반 EDA 리포트
- 데이터 프로파일링 리포트
- 전처리 및 마스터 테이블 생성 리포트

### 1. 데이터 점검

- 데이터셋을 불러오고 상위 5개 행과 하위 5개 행을 출력합니다.
- `df.info()`를 사용하여 컬럼 타입, non-null 개수 등 기본 정보를 확인합니다.
- 전체 행 수와 열 수를 보고합니다.
- 중복 레코드 여부를 확인하고 중복 개수를 보고합니다.

### 2. 기술통계

- 수치형 변수와 범주형 변수 모두에 대해 기술통계를 수행합니다.
- 각 기술통계 요약에 대해 1,000자 이상의 상세한 한국어 분석 보고서를 작성합니다.
- 보고서에는 분포, 중심 경향, 눈에 띄는 패턴, 이상치, 잠재적인 비즈니스 의미를 포함합니다.

### 3. 범주형 데이터 분석

- 범주형 변수에 대해 빈도 그래프를 생성합니다.
- 고유값이 많은 변수는 상위 30개 항목만 출력하고 시각화합니다.

### 4. 텍스트 데이터 분석

텍스트 데이터가 있는 경우에만 수행합니다.

- 시간이 오래 걸리는 형태소 분석은 수행하지 않습니다.
- 키워드 추출에는 TF-IDF를 사용합니다.
- 빈도 또는 점수 기준 상위 30개 키워드를 출력합니다.
- 키워드 빈도를 시각화합니다.
- 키워드 결과는 반드시 표로도 제공합니다.

### 5. 고급 시각화

- 최소 10개 이상의 서로 다른 시각화를 생성합니다.
- 일변량, 이변량, 다변량 분석을 다양하게 포함합니다.
- **필수 표 포함**: 모든 시각화마다 관련 교차표, 피벗테이블 또는 기술통계표를 함께 제공합니다.
- **해석 포함**: 모든 시각화마다 50자 이상의 한국어 해석을 작성합니다. 해석에는 해당 차트가 무엇을 보여주는지, 어떤 의미가 있는지 설명합니다.

## 최종 리포트 요구사항

- 모든 설명, 해석, 리포트 내용은 한국어로 작성합니다.
- 최종 결과물은 `eda_report.md`와 같은 단일 마크다운 파일이어야 합니다.
- 리포트는 논리적인 구조를 가져야 합니다. 예시는 다음과 같습니다.

  1. 서론
  2. 데이터 개요
  3. 상세 기술통계
  4. 시각화 분석
  5. 결론

## 프로젝트형 EDA 추가 지침

기본 EDA를 수행하되, 단순히 데이터 전체를 넓게 훑는 것에 그치지 말고 프로젝트 주제와 분석 목적에 맞게 분석을 구성합니다.

### 0. 분석 목적 및 분석 단위 정의

EDA를 시작하기 전에 먼저 아래 내용을 정리합니다.

- 분석 목적은 무엇인가?
- 분석 단위는 무엇인가?
  - 예: 고객 단위, 주문 단위, 셀러 단위, 상품 단위, 지역 단위
- 핵심 분석 질문은 무엇인가?
- 어떤 지표를 중심으로 분석할 것인가?
- 현재 데이터로 해당 질문을 검증할 수 있는가?

분석 목적과 분석 단위가 명확하지 않은 경우, 무리하게 리포트를 작성하지 말고 먼저 가능한 분석 방향을 제안합니다.

### 관계형 데이터셋 처리

여러 개의 CSV 또는 테이블로 구성된 데이터셋은 단일 파일처럼 바로 EDA하지 않습니다.

먼저 다음을 수행합니다.

- 각 파일의 역할 파악
- 각 테이블의 행 수, 열 수, 주요 컬럼 확인
- 주요 키 컬럼 확인
  - 예: order_id, customer_id, seller_id, product_id
- 테이블 간 관계 정리
- 분석 목적에 필요한 테이블 선별
- 조인 전후 행 수 변화 검증
- 분석 단위에 맞는 마스터 테이블 또는 요약 테이블 생성

조인 후 행 수가 비정상적으로 증가하면 원인을 확인하고 리포트에 명시합니다.

### 지표 정의

분석에 사용하는 주요 지표는 반드시 정의합니다.

각 지표에 대해 아래 내용을 작성합니다.

- 지표명
- 계산식
- 사용 컬럼
- 집계 단위
- 해석 방법
- 주의사항

예시:

- total_sales = seller_id별 price 합계
- on_time_rate = 실제 배송완료일이 예상 배송일 이하인 주문 비율
- freight_ratio = freight_value / price

### 데이터 검증 및 산출물 검증

분석 결과를 생성한 뒤 아래 항목을 검증합니다.

- 조인 전후 행 수가 논리적으로 맞는가?
- 주요 키 컬럼의 고유값 수가 예상과 맞는가?
- 집계 단위가 분석 목적과 일치하는가?
- 주요 지표의 계산식이 리포트에 명시되어 있는가?
- 그래프의 값과 표의 값이 일치하는가?
- 결론이 실제 지표에 의해 뒷받침되는가?
- 데이터에 없는 내용을 추정하거나 단정하지 않았는가?

### 해석 원칙

EDA는 탐색적 분석이므로 인과관계를 단정하지 않습니다.

피해야 할 표현:

- A 때문에 B가 발생했다
- A가 B의 원인이다
- A가 B를 증명한다

권장 표현:

- A와 B 사이에 차이가 관찰된다
- A 그룹에서 B 지표가 더 높게 나타난다
- 이러한 경향은 추가 검증이 필요하다
- 본 분석에서는 관련성 또는 차이를 확인하는 수준으로 해석한다

### 분석 한계

최종 리포트에는 반드시 분석 한계 섹션을 포함합니다.

아래 내용을 검토합니다.

- 데이터 수집 기간의 한계
- 결측치와 이상치 처리 한계
- 조인 과정에서 발생할 수 있는 데이터 중복 문제
- 분석 단위와 원본 데이터 단위가 다른 경우의 한계
- 인과관계가 아닌 탐색적 분석이라는 점
- 외부 시장이나 다른 국가에 일반화하기 어려운 점

### 비즈니스 액션플랜

주요 인사이트마다 실행 가능한 액션플랜을 제안합니다.

각 액션플랜은 아래 형식을 따릅니다.

- 대상
- 관찰된 문제
- 근거 지표
- 제안 액션
- 기대 효과
- 추적 KPI
- 추가 검증 필요 여부

### 시각화 구성 원칙

시각화는 단순히 개수를 채우기 위해 생성하지 않습니다.  
분석 목적과 연결되는 그래프를 우선 생성합니다.

각 시각화는 아래 목적 중 하나에 해당해야 합니다.

- 데이터 구조 이해
- 주요 변수 분포 확인
- 그룹 간 비교
- 시간 흐름 확인
- 변수 간 관계 확인
- 이상치 확인
- 비즈니스 의사결정 근거 제시

### 시간/시계열 데이터 분석

날짜 또는 시간 컬럼이 있는 경우 시간/시계열 분석을 선택적으로 수행합니다.

예시 컬럼:

- 주문일
- 결제일
- 승인일
- 출고일
- 배송완료일
- 예상배송일
- 리뷰작성일

가능한 분석:

- 월별 주문 추이
- 요일별 주문 분포
- 시간대별 주문 패턴
- 월별 매출 추이
- 배송 소요일 변화
- 특정 기간의 급증/감소 여부

단, 시간/시계열 분석은 프로젝트 주제와 직접 관련 있을 때만 핵심 분석으로 사용합니다.  
프로젝트 주제가 셀러 비교, 고객 세그먼트, 상품 분석 등인 경우 시간 분석은 보조 분석 또는 참고 섹션으로 배치합니다.

### 최종 리포트 권장 구조

최종 리포트는 아래 구조를 권장합니다.

1. 분석 주제
2. 분석 목적
3. 데이터 개요
4. 분석 단위 및 지표 정의
5. 데이터 프로파일링 결과
6. 전처리 및 조인 기준
7. 주요 EDA 결과
8. 핵심 인사이트
9. 비즈니스 액션플랜
10. 분석 한계
11. 검증 체크리스트

### 프로젝트형 EDA 추가 규칙

- 단순한 일반 EDA만 수행하지 않습니다. 반드시 주어진 프로젝트 주제와 분석 목적에 맞게 분석을 구성합니다.
- 프로젝트 주제가 제공된 경우, 해당 주제와 직접 관련 있는 변수와 시각화를 우선합니다.
- 데이터 구조상 주제를 검증하기 어렵다면 그 이유를 명확히 설명하고, 가능한 대체 분석 질문을 제안합니다.
- 여러 데이터셋이 제공된 경우 병합 전에 각 데이터셋을 개별적으로 프로파일링합니다.
- 조인 후에는 행 수 변화와 키 중복 여부를 반드시 검증합니다.
- EDA 결과를 인과관계로 단정하지 않습니다. 관찰된 패턴, 차이, 경향으로 표현합니다.
- 최종 리포트에는 분석 결과뿐 아니라 비즈니스 의미와 실행 가능한 액션플랜을 포함합니다.

## 프로젝트 주제 기반 리포트 보완 규칙

프로젝트 주제가 명확히 제공된 경우, 일반 EDA 리포트보다 프로젝트 주제에 직접 연결되는 분석을 우선합니다.

### 1. 리포트 유형 명시

최종 리포트 초반에 본 리포트가 어떤 유형인지 명시합니다.

- 일반 EDA 리포트
- 데이터 프로파일링 리포트
- 프로젝트 주제 기반 EDA 리포트
- 전처리 및 마스터 테이블 생성 리포트
- 그룹 비교 분석 리포트

프로젝트 주제가 제공된 경우, 리포트 제목과 본문은 반드시 해당 주제를 반영해야 합니다.

예시:

- 부적절한 제목: `Olist 이커머스 데이터 종합 분석 리포트`
- 적절한 제목: `Olist 마켓플레이스 Top 20% vs Bottom 20% 셀러 특성 비교 분석 리포트`

### 2. 분석 주제와 무관한 일반 EDA 축소

일반 EDA 항목은 데이터 이해를 위한 참고 자료로 사용할 수 있지만, 프로젝트 주제와 직접 관련이 없다면 메인 분석으로 배치하지 않습니다.

예를 들어 프로젝트 주제가 `seller_id 기준 Top 20% vs Bottom 20% 셀러 비교`라면 아래 항목은 보조 또는 부록으로 배치합니다.

- 월별 주문 추이
- 요일별 주문 분포
- 전체 결제수단 분포
- 전체 고객 지역 분포
- 전체 상품 카테고리 빈도
- 일반 TF-IDF 키워드 분석

메인 분석은 반드시 프로젝트 주제와 직접 관련된 지표와 시각화를 중심으로 구성합니다.

### 3. 그룹 비교 분석 강화

분석 주제에 상위/하위 그룹, 세그먼트, 고객군, 상품군, 지역군 비교가 포함된 경우, 그룹 비교를 리포트의 핵심 구조로 배치합니다.

각 그룹 비교 분석에는 아래 내용을 포함합니다.

- 그룹 정의 기준
- 그룹별 샘플 수
- 그룹별 핵심 지표 비교표
- 그룹별 시각화
- 그룹 간 차이에 대한 해석
- 차이 해석 시 주의사항
- 보조 기준 또는 민감도 분석

예시:

- Top 20% / Bottom 20% 그룹 정의 기준
- 전체 기준과 최소 주문 수 기준 비교
- Top 20% vs Bottom 20% 핵심 비교표
- 그룹별 리뷰, 배송, 가격, 배송비율, 카테고리 수 비교

### 4. 핵심 비교표 필수 작성

그룹 비교 분석을 수행하는 경우, 최종 리포트에 핵심 비교표를 반드시 포함합니다.

비교표에는 아래 항목을 포함합니다.

- 그룹명
- 그룹별 데이터 수
- 주요 성과 지표
- 주요 품질 지표
- 주요 비용/가격 지표
- 주요 운영 지표
- 그룹 간 차이 또는 비율

예시:

| 지표 | Top 20% | Bottom 20% | 해석 |
|---|---:|---:|---|
| 평균 매출 |  |  |  |
| 평균 주문 수 |  |  |  |
| 평균 리뷰 점수 |  |  |  |
| 예상일 내 도착비율 |  |  |  |
| 평균 상품 가격 |  |  |  |
| 배송비율 |  |  |  |
| 카테고리 수 |  |  |  |

### 5. 보조 분석 및 민감도 분석

특정 그룹에 극단값, 소수 표본, 판매 건수 1건짜리 데이터 등이 포함될 수 있는 경우 보조 분석을 수행합니다.

예시:

- 전체 셀러 기준 분석
- `order_count >= 5` 셀러 기준 분석
- 이상치 제거 전후 비교
- 특정 기간 제외 전후 비교
- 결측치 포함/제외 기준 비교

보조 분석의 목적은 결과가 특정 조건에 과도하게 의존하는지 확인하는 것입니다.

최종 리포트에는 기본 분석과 보조 분석의 차이를 설명합니다.

### 6. 표와 해석 불일치 검증

최종 리포트를 작성한 뒤, 모든 표와 해석이 일치하는지 검증합니다.

아래 사항을 반드시 확인합니다.

- 표에서 더 높은 값으로 표시된 그룹이 해석에서도 동일하게 설명되었는가?
- 그래프와 표의 수치가 서로 일치하는가?
- 평균, 비율, 합계가 서로 혼동되지 않았는가?
- %와 소수 표현이 혼용되어 오해를 만들지 않는가?
- 해석이 실제 수치와 반대로 작성되지 않았는가?

예시:

- 표에서 Bottom 20%의 `on_time_rate`가 Top 20%보다 높다면, 해석에서도 이를 정확히 반영해야 합니다.
- 수치상 차이가 작다면 “큰 차이”라고 표현하지 않습니다.

### 7. 과장 표현 및 인과 표현 자동 점검

EDA 리포트에서는 탐색적 분석 결과를 인과관계로 단정하지 않습니다.

아래 표현은 사용을 피합니다.

- 증명한다
- 원인이다
- 반드시 필요하다
- 직접적으로 영향을 미친다
- ~ 때문에 발생했다
- 강력하게 뒷받침한다
- 직결된다
- 확신한다

아래 표현을 우선 사용합니다.

- 관찰된다
- 경향을 보인다
- 차이가 나타난다
- 가능성을 시사한다
- 추가 검증이 필요하다
- 본 데이터에서는 ~로 해석할 수 있다
- 단정하기는 어렵다

최종 리포트 작성 후 인과관계 단정 표현이 포함되어 있는지 점검하고, 필요한 경우 완화된 표현으로 수정합니다.

### 8. 지표 정의 표 필수화

프로젝트형 EDA에서는 주요 지표 정의표를 반드시 작성합니다.

각 지표에 대해 아래 내용을 포함합니다.

- 지표명
- 계산식
- 사용 컬럼
- 집계 단위
- 해석 방법
- 주의사항

예시:

| 지표명 | 계산식 | 집계 단위 | 해석 | 주의사항 |
|---|---|---|---|---|
| total_sales | seller_id별 price 합계 | seller_id | 셀러 상품 매출 | 배송비는 제외 |
| on_time_rate | 실제 배송완료일 <= 예상 배송일 비율 | seller_id | 예상일 내 도착비율 | delivered 주문 중심 |
| freight_ratio | freight_value / price | order item 또는 seller_id | 상품가 대비 배송비 부담 | price가 0이면 제외 |
| avg_review_score | review_score 평균 | seller_id | 고객 만족도 대리 지표 | 리뷰는 주문 단위임 |

### 9. 분석 한계의 구체화

분석 한계는 일반 문장으로만 작성하지 않고, 실제 데이터 구조와 연결하여 작성합니다.

아래 사항을 검토합니다.

- 분석 단위와 원본 데이터 단위가 다른 경우
- 조인 과정에서 행 수가 증가하는 경우
- 리뷰, 결제, 배송 데이터의 단위가 서로 다른 경우
- 특정 그룹의 표본 수가 너무 적은 경우
- 일부 기간 데이터가 불완전한 경우
- 외부 시장이나 다른 국가에 일반화하기 어려운 경우

예시:

- 리뷰는 `order_id` 단위로 제공되므로, 한 주문에 여러 셀러 상품이 포함된 경우 리뷰 점수를 특정 셀러에게 완전히 귀속하기 어렵습니다.
- 하위 20% 셀러에는 판매 건수가 매우 적은 셀러가 포함될 수 있어 평균값 해석에 주의가 필요합니다.
- Olist 데이터는 브라질 이커머스 데이터이므로 국내 이커머스 시장에 직접 일반화하기 어렵습니다.

### 10. 최종 인사이트 압축

최종 결론은 너무 많은 인사이트를 나열하지 않습니다.

핵심 인사이트는 3~5개로 압축합니다.

각 인사이트는 아래 형식으로 작성합니다.

- 인사이트
- 근거 지표
- 해석
- 비즈니스 의미
- 추가 검증 필요 여부

예시:

- 인사이트: Top 20% 셀러는 Bottom 20% 셀러보다 평균 주문 수와 상품 가격이 높게 관찰됩니다.
- 근거 지표: `total_orders`, `avg_price`
- 해석: 매출 격차는 단순한 리뷰 차이보다 거래 규모와 가격 구조 차이와 더 밀접하게 관찰됩니다.
- 비즈니스 의미: 하위 셀러의 성장을 위해 초기 주문 확보와 가격·배송비 구조 개선이 필요할 수 있습니다.
- 추가 검증 필요 여부: 고객 전환율 또는 노출 데이터가 있다면 추가 검증이 필요합니다.

### 11. 액션플랜의 KPI 연결

비즈니스 액션플랜은 단순 제안으로 끝내지 않고, 반드시 추적 KPI와 연결합니다.

각 액션플랜은 아래 형식을 따릅니다.

| 대상 | 관찰된 문제 | 근거 지표 | 제안 액션 | 기대 효과 | 추적 KPI |
|---|---|---|---|---|---|

예시:

| 대상 | 관찰된 문제 | 근거 지표 | 제안 액션 | 기대 효과 | 추적 KPI |
|---|---|---|---|---|---|
| 하위 셀러 | 배송비율이 높음 | freight_ratio | 묶음배송/배송비 보조 | 구매 부담 완화 | freight_ratio, order_count |
| 저평점 셀러 | 낮은 리뷰 점수 | avg_review_score | CS 가이드 제공 | 고객 만족도 개선 | avg_review_score, low_review_rate |

### 12. 최종 리포트 재배치 원칙

프로젝트형 EDA 리포트는 아래 순서를 우선 권장합니다.

1. 분석 주제
2. 분석 목적
3. 분석 단위 및 지표 정의
4. 데이터 프로파일링 요약
5. 전처리 및 조인 기준
6. 그룹 정의
7. 핵심 비교표
8. 주요 EDA 결과
9. 보조 분석 또는 민감도 분석
10. 핵심 인사이트
11. 비즈니스 액션플랜
12. 분석 한계
13. 검증 체크리스트
14. 부록: 일반 EDA 또는 참고 분석

일반 EDA, TF-IDF, 월별 추이, 요일별 패턴 등 프로젝트 주제와 직접 관련이 낮은 분석은 부록으로 배치합니다.

### 13. 최종 검증 체크리스트 추가

최종 리포트 작성 후 아래 항목을 추가로 검증합니다.

1. [ ] 리포트 제목이 프로젝트 주제를 반영하는가?
2. [ ] 분석 단위가 명확히 정의되었는가?
3. [ ] 주요 지표 정의표가 포함되었는가?
4. [ ] 그룹 비교 기준이 명확한가?
5. [ ] 핵심 비교표가 포함되었는가?
6. [ ] 표와 해석이 서로 일치하는가?
7. [ ] 수치가 반대로 해석된 부분이 없는가?
8. [ ] 일반 EDA가 메인 분석을 방해하지 않는가?
9. [ ] 보조 분석 또는 민감도 분석이 필요한 경우 수행되었는가?
10. [ ] 인과관계 단정 표현을 피했는가?
11. [ ] 핵심 인사이트가 3~5개로 정리되었는가?
12. [ ] 액션플랜이 근거 지표와 KPI에 연결되었는가?
13. [ ] 분석 한계가 데이터 구조와 연결되어 구체적으로 작성되었는가?
